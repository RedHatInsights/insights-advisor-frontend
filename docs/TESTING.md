## Overview

The Insights Advisor Frontend has a comprehensive test suite using **Jest** for unit/component tests and **Cypress** for integration/E2E tests.

**Philosophy**: Tests should verify user-visible behavior, not implementation details. Mock at system edges (HTTP, user input), not internal components.

**Test Quality Improvement Project**: Started February 2026 to transform the test suite from snapshot-heavy to behavioral testing.

## Testing Patterns

### ✅ Good Patterns (Follow These)

#### 1. Mock at System Edges

```javascript
// ✅ GOOD: Mock HTTP layer
import { instance } from 'Services/Recs';
jest.mock('Services/Recs');

instance.mockResolvedValue({
  meta: { count: 10 },
  data: [{ rule_id: 'test-rule' }],
});

render(<Component />);
expect(screen.getByText('test-rule')).toBeInTheDocument();
```

#### 2. Test User-Visible Behavior

```javascript
// ✅ GOOD: Test what users see and do
const user = userEvent.setup();
render(<SystemsTable />);

await user.click(screen.getByRole('button', { name: /filter/i }));
expect(screen.getByRole('menu')).toBeVisible();

await user.click(screen.getByRole('menuitem', { name: /security/i }));
expect(screen.getByText('Security recommendations')).toBeInTheDocument();
```

#### 3. Use React Testing Library Queries

```javascript
// ✅ GOOD: Query hierarchy (best → worst)
screen.getByRole('button', { name: /submit/i })  // Best
screen.getByLabelText('Email')                   // Good
screen.getByText('Submit')                       // OK
screen.getByTestId('submit-btn')                 // Last resort

// ❌ BAD: DOM queries
container.querySelector('.submit-button')        // Never!
```

#### 4. Cypress Component Tests

```javascript
// ✅ GOOD: Cypress component test pattern
import { mount } from '@cypress/react';
import RulesTable from './RulesTable';

describe('RulesTable', () => {
  beforeEach(() => {
    // Mock API at network layer
    cy.intercept('GET', '/api/insights/v1/rule/*', {
      statusCode: 200,
      body: { data: mockRules },
    });
  });

  it('filters rules by category', () => {
    mount(<RulesTable />);
    
    cy.get('[data-ouia-component-id="category-filter"]').click();
    cy.get('[data-ouia-component-id="Security"]').click();
    
    cy.contains('Security rule 1').should('be.visible');
    cy.contains('Performance rule 1').should('not.exist');
  });
});
```

### ❌ Bad Patterns (Avoid These)

#### 1. Don't Mock Internal Components

```javascript
// ❌ BAD: Mocking internal components
jest.mock('./InternalComponent', () => ({
  InternalComponent: jest.fn(() => <div>Mocked</div>),
}));

render(<Parent />);
expect(InternalComponent).toHaveBeenCalledWith({ prop: 'value' });
// This tests implementation, not behavior!
```

#### 2. Don't Test Implementation Details

```javascript
// ❌ BAD: Testing internal state structure
const { result } = renderHook(() => useMyHook());
expect(result.current.internalState).toEqual({ ... });

// ✅ GOOD: Test observable outcomes
const { result } = renderHook(() => useMyHook());
act(() => result.current.doSomething());
expect(result.current.publicValue).toBe('expected');
```

#### 3. Don't Use Snapshots for Behavior

```javascript
// ❌ BAD: Snapshot test (removed from codebase)
expect(wrapper).toMatchSnapshot();

// ✅ GOOD: Behavioral test
expect(screen.getByRole('heading')).toHaveTextContent('Expected Title');
expect(screen.getByRole('button')).toBeEnabled();
```

#### 4. Don't Break Tests on Refactoring

```javascript
// ❌ BAD: Brittle test tied to structure
expect(wrapper.find('div').at(2).find('span').text()).toBe('test');

// ✅ GOOD: Test based on user perspective
expect(screen.getByText('test')).toBeInTheDocument();
```

## Testing Principles

### Core Principles

**✅ Good Tests Should:**
1. **Test behaviors, not implementation details**
   - Focus on what users see and do
   - Survive refactoring without modification
   
2. **Mock at system edges**
   - HTTP layer (axios, RTK Query, fetch)
   - User input (userEvent, cy.click)
   - Time (jest.useFakeTimers, cy.clock)
   - NOT internal components or functions
   
3. **Provide deployment confidence**
   - If tests pass, feature works
   - If feature breaks, tests fail
   - No false positives or negatives
   
4. **Be maintainable**
   - Clear test names describing behavior
   - No magic numbers or unexplained values
   - Use constants from Messages.js for text assertions

**❌ Bad Tests:**
1. Mirror code structure directly (testing internals)
2. Mock internal dependencies and layers
3. Break when refactoring non-behavioral changes
4. Use snapshots instead of explicit assertions
5. Test coverage without confirming business logic

### Testing Philosophy

**"Test like a user, not like a developer"**

Users don't care about:
- Component props
- Internal state structure
- Function call counts
- Implementation details

Users DO care about:
- What they see on screen
- Whether buttons work
- If data loads correctly
- Error messages when things fail

### Cypress vs Jest Quality

**Key Insight**: Cypress tests are 92% good quality, Jest tests were 49% good quality at start.

**Why?**
- Cypress forces system-edge mocking (`cy.intercept`)
- Cypress encourages user perspective (`cy.contains`, `cy.click`)
- Cypress makes it hard to test implementation details

**Strategy**: Apply Cypress testing patterns to Jest tests!

## Common Testing Pitfalls

### 1. URL Parameter Testing in Cypress

**Problem**: MemoryRouter doesn't set `window.location.search`

**Solution**: Use `window.history.pushState()` before mounting

```javascript
// ❌ BAD: MemoryRouter alone doesn't work
mount(
  <MemoryRouter initialEntries={['/recommendations?incident=true']}>
    <RulesTable />
  </MemoryRouter>
);
// paramParser() still reads empty string from window.location.search!

// ✅ GOOD: Set browser URL first
cy.window().then((win) => {
  win.history.pushState({}, '', '/recommendations?incident=true');
});
mount(<RulesTable />);
// Now paramParser() reads the correct URL params
```

**Why**: `paramParser()` in `Common/Tables.js` uses `window.location.search`, not React Router state.

### 2. Axios Response Interceptor

**Problem**: Assuming standard axios response structure

**Reality**: `useAxiosWithPlatformInterceptors` auto-unwraps `response.data`

```javascript
// ❌ BAD: Double unwrap
const response = await axios.get(url);
const data = response?.data?.playbook_count;

// ✅ GOOD: Already unwrapped
const response = await axios.get(url);
const data = response?.playbook_count;
```

**Affects**: All axios calls in components. See Inventory.js tests for examples.

### 3. Jest ESM Package Transformation

**Problem**: ESM packages throw "Cannot use import statement outside a module"

**Solution**: Add to `transformIgnorePatterns` in `package.json`

```json
{
  "jest": {
    "transformIgnorePatterns": [
      "node_modules/(?!(@redhat-cloud-services|bastilian-tabletools|p-all|p-map|aggregate-error|indent-string|clean-stack)/)"
    ]
  }
}
```

**Currently Transformed**: `@redhat-cloud-services`, `bastilian-tabletools`, `p-all`, `p-map`, `aggregate-error`, `indent-string`, `clean-stack`

### 4. Filter Value Types

**Issue**: Filters expect specific value types from URL parameters

**Correct Types**:
- **Category filters**: Numeric IDs (`2` = security, `3` = availability)
- **Boolean filters**: String arrays (`['true']`, `['false']`)
- **Numeric filters**: String arrays (`['4']`, `['3']`, `['2']`, `['1']`)

**Common Bug**: URL params like `?incident=true` (string) passed to CheckboxFilter expecting array

**Fix Pattern**:
```javascript
value: Array.isArray(filters.incident)
  ? filters.incident
  : filters.incident
    ? [String(filters.incident)]
    : []
```

See `filterArraySafety.test.js` for comprehensive tests.

### 5. PatternFly Charts in Jest

**Problem**: PatternFly Charts cause `Cannot read properties of undefined (reading 'replace')` in Jest

**Solution**: Exclude from coverage or test with Cypress instead

**Excluded Components**:
- `TotalRiskCard.js` (uses PatternFly Charts)
- `IopOverviewDashbarCard.js` (IOP-specific, low priority)
- Export/PDF components (complex DOM manipulation)

**Strategy**: Test these with Cypress component tests instead of Jest.

### 6. RTK Query Testing

**Simple Approach** (Recommended):
```javascript
// Just test the API structure
expect(api.endpoints.getRule).toBeDefined();
expect(api.endpoints.getRule.query).toBeDefined();
```

**Complex Approach** (Only when needed):
```javascript
// Full integration test with store
const store = initStore();
await store.dispatch(api.endpoints.getRule.initiate('rule-123'));
// ... test async behavior
```

**When to use complex**: Only when testing data transformations in `transformResponse`.

## Testing Checklist

### When Adding a New Component

- [ ] Create co-located `.test.js` file
- [ ] Test main user interactions (clicks, inputs)
- [ ] Test data rendering (success state)
- [ ] Test loading state
- [ ] Test error state
- [ ] Test empty state (if applicable)
- [ ] Mock API at HTTP layer (not internal components)
- [ ] Use `screen.getByRole()` queries (accessibility!)
- [ ] Run coverage: `npm run test:coverage`

### When Adding a New Feature

- [ ] Update existing tests if behavior changed
- [ ] Add new test cases for new functionality
- [ ] Test both success and failure paths
- [ ] Test edge cases (empty data, invalid input)
- [ ] Consider Cypress integration test for full flow
- [ ] Update documentation if architecture changes

### When Fixing a Bug

- [ ] Write test that reproduces the bug (should fail)
- [ ] Fix the bug
- [ ] Verify test now passes
- [ ] Add regression test to prevent recurrence
- [ ] Document the bug in test comments if subtle
